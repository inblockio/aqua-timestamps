### 3. Implement the M0 surface (1 h)

- `GET /health` → `200` with JSON `{"status":"ok","uptime_secs":<int>}`.
- `GET /` → `200 text/html` with a single-file landing page: name of
  the service, one-paragraph what-it-is, two links (`/health` and
  `/.well-known/aqua-identity` — the latter will 404 at M0, that's
  fine, the link is for M1).
- `tracing_subscriber` for structured logs, `RUST_LOG` honored.
- `--config <path>` flag, loads `config.toml` with at minimum
  `[server] listen = "0.0.0.0:8080"`.

Done when: `cargo run -- --config config.toml` serves both endpoints
locally, and `tests/smoke_health.rs` passes.

### 4. Dockerize (30 min)

Multi-stage Dockerfile:

```dockerfile
FROM rust:1-bookworm AS builder
WORKDIR /build
COPY . .
RUN cargo build --release --bin aqua-timestamp

FROM debian:bookworm-slim
RUN useradd -r -u 10001 -s /sbin/nologin aqua
COPY --from=builder /build/target/release/aqua-timestamp /usr/local/bin/
USER aqua
EXPOSE 8080
HEALTHCHECK --interval=10s --timeout=3s --start-period=15s \
  CMD ["/usr/local/bin/aqua-timestamp", "healthcheck"] || \
  CMD wget -qO- http://localhost:8080/health || exit 1
CMD ["aqua-timestamp", "--config", "/etc/aqua-timestamp/config.toml"]
```

Done when: `docker buildx build -t aqua-timestamp:m0 deploy/` succeeds,
`docker run --rm -p 8080:8080 aqua-timestamp:m0` serves `/health` on
localhost, final image is under 100 MB.

### 5. Ship to the server (15 min)

No GHCR yet. Save and load:

```sh
docker save aqua-timestamp:m0 | \
  ssh -i ~/.ssh/timestamp_deploy_ed25519 root@timestamp.inblock.io \
    'docker load'
```

Create the deploy dir on the server. Recommend `/root/timestamp/` for
M0 (simplest; a dedicated `timestamp` user can come later if desired):

```sh
ssh -i ~/.ssh/timestamp_deploy_ed25519 root@timestamp.inblock.io \
  'mkdir -p /root/timestamp'
scp -i ~/.ssh/timestamp_deploy_ed25519 \
  deploy/docker-compose.yml root@timestamp.inblock.io:/root/timestamp/
```

The compose must declare `portal-net` as `external: true` so we attach
to the existing network. Service name `timestamp` so Caddy reaches us
as `timestamp:8080`.

Bring it up:

```sh
ssh -i ~/.ssh/timestamp_deploy_ed25519 root@timestamp.inblock.io \
  'cd /root/timestamp && docker compose up -d'
```

Done when: `docker ps` on the server shows `timestamp` healthy on
`portal-net`.

### 6. Wire into Caddy (10 min)

Append (don't replace) to `/home/portal/portal/Caddyfile`:

```caddy
timestamp.inblock.io {
    reverse_proxy timestamp:8080
}
```

Pattern that won't surprise the portal owner: keep the existing block
intact, add the new one below it.

Reload Caddy:

```sh
ssh -i ~/.ssh/timestamp_deploy_ed25519 root@timestamp.inblock.io \
  'docker exec portal-caddy-1 caddy reload --config /etc/caddy/Caddyfile'
```

Caddy auto-provisions Let's Encrypt on first external request. DNS
already resolves, so the cert dance happens immediately.

## What to avoid

Catalogued so the next session doesn't re-litigate.

- **Don't reimplement SDK primitives.** Use `aqua-rs-sdk` for Merkle,
  Object, Signature, templates. M0 doesn't need them, but the deps go
  in at M0.
- **Don't use Ed25519 anywhere.** The shipped design-spec.md says
  Ed25519; ignore it. The SDK and the reference identity at
  `https://aquafire.inblock.io/.well-known/aqua-identity` are
  authoritative: secp256k1 + EIP-191 + SIWE.
- **Don't `docker build` on the server.** Disk on the deploy box is
  fine right now (110 GB free), but the established pattern is build
  locally, ship the image.
- **Don't set up GH Actions yet.** Owner explicitly deferred CI until
  after M0.
- **Don't refactor the portal's Caddyfile to use imports yet.** Append
  a site block. The refactor is a separate, future decision.
- **Don't generate a new wallet.** The service key already exists in
  the keyring. Re-generating would invalidate any Sepolia funding the
  owner has sent.
